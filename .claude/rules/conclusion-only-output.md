# 成果物は結論だけを書く（過程・対比・自己言及を残さない）

## 背景

レビュー・PRD・Issue・PRコメント・設計 doc など**対外的な成果物**で、結論に至る「過程」を本文に混ぜてしまう問題がある。読み手が必要としているのは確定した結論だけで、「誰がどう言って・どう検証して・どう変わったか」という制作の経緯は不要なノイズになる。

特に AI は、自分の検証プロセス（別ツールの意見、敵対的検証、当初の仮説と訂正）を成果物に書き残しがち。これは「丁寧さ」ではなく冗長で、読み手の判断を遅らせる。

## 原則

**成果物には、確定した結論とその根拠だけを書く。そこに至る過程は書かない。**

「どう調べたか」「最初はこう考えたが訂正した」「ツールAはこう言ったがツールBで覆った」は、自分の作業ログであって成果物の中身ではない。本文に残さず、必要なら口頭・チャット側の報告に留める。

## 書かないもの（過程・対比・自己言及）

| 書きがちなもの | なぜダメか | どうするか |
|---|---|---|
| 「codex は X と言ったが、検証で Y と分かった」 | 読み手に X は不要。確定は Y だけ | **Y だけを書く** |
| 「敵対的検証側は Important 評価だが、私は Critical とする」 | 評価が割れた過程は内部事情 | 最終判定（Critical）と理由だけ書く |
| 「当初は A と考えたが、B に訂正した」 | 捨てた仮説 A は読み手に無関係 | 結論 B だけを書く |
| 「5次元×敵対的検証ワークフロー + codex review の統合により…」 | どう作ったかの自慢は中身ではない | 手法の自己説明を削る |
| 「〜を確認した」「〜を調べた結果」 | 調査の過程描写 | 確認できた事実そのものを書く |
| 「念のため」「一応」「まず〜してから」 | 作業手順のナレーション | 結論に直接効く文だけ残す |

## 書くもの（結論と根拠）

- **確定した結論**: 判定・症状・原因・推奨アクション
- **その結論を成り立たせる根拠**: コードの該当箇所、再現条件、数字。これは「過程」ではなく「結論の裏付け」なので書く
- **読み手が次に取るべき行動**: 修正案、確認すべき点

> 区別の基準: 「読み手が判断・行動するのに要るか」で切る。要らない経緯（誰が言った・どう検証した・何を捨てた）は過程。要る裏付け（どこのコードが・どういう条件で）は根拠。

## Before / After（実例）

❌ Before（過程・対比が混ざっている）:
> 症状の正確な記述: codex は「500になる」と推測したが、敵対的検証で500ではなく「HTTP 200を返しつつ保存が消えるサイレントなロスト更新」であることを確認した。

✅ After（結論だけ）:
> 症状は HTTP 200 を返しつつ、同一リクエストの主処理書き込みがサイレントに巻き戻る（例外は出ない）。ユーザーには成功に見えて保存が消える。

❌ Before（自己言及の手法説明）:
> このレビューは 5次元×敵対的検証ワークフロー + codex review の統合により生成しました。

✅ After（手法の自己説明を削る）:
> 手法・検証プロセスの説明は本文に書かない。AI 生成であることの開示が必要なら、過程ナレーション（「実コードで検証済み」等）ではなく、末尾の定型フッター（例: `Generated by Claude Code`）に留める。

## セルフチェック（成果物を書き終えたら）

1. **過程チェック**: 「誰がどう言って・どう検証して・どう変わったか」という制作の経緯が本文に混ざっていないか？ → 削って結論だけにする
2. **対比チェック**: 「A は〜だが B は〜」という、捨てた選択肢・覆った仮説との対比が残っていないか？ → 確定した側だけ残す
3. **自己言及チェック**: 自分の調査手順・使ったツール・ワークフロー構成を説明していないか？ → 削る
4. **根拠は残す**: 結論の裏付け（コード位置・再現条件・数字）まで削っていないか？ これは過程ではないので残す
