# コードコメントのルール

## 原則: Evergreen Comments（時間が経っても腐らないコメント）

コードに残すコメントは **Evergreen comments** / **Timeless documentation** / **Intent-only comments** と呼ばれる考え方に従う。骨子は次の 2 つ:

- コードに書けなかった情報だけを補う（実装で表現できることはコメントで重複させない）
- 履歴ではなく今の意図を書く（時間が経つと腐る情報はコードに混ぜない）

### 参考プラクティス

- Martin Fowler: "Good comments express intent at the level you cannot express in code"
- John Ousterhout "A Philosophy of Software Design": コメントは「コードに書けなかった情報」だけを補う
- Google C++ Style Guide: 実装コメントはコードの現在の振る舞いを説明するもので、変更履歴を書く場所ではない

### 判断基準

> 迷ったら「履歴を知らない新しいメンバーが読んだときに意味が通るか？」で判断する。通らなければ履歴情報なので削る。

この基準を通った「現在こうなっている WHY」だけを残す。WHAT（何をしているか）はコード自体が語るので書かない。

## 書くもの: 現在の WHY

コードを初見で理解するために必要な「なぜ今こうなっているか」を残す。

- ビジネス上の制約（例: 規約で〇〇できない）
- 設計判断の意図（例: 同期負荷を避けるため非同期化）
- 気づきにくい副作用・前提
- 削除判断の材料（例: 〜が不要になれば削除可能）

### 該当箇所

- **公開関数** (`export`): JSDoc で目的・背景・使用箇所
- **非自明なロジック**: 「なぜこの方法か」「なぜこの順序か」
- **ワークアラウンド**: 理由と、いつ不要になるか
- **型・インターフェース**: 何を表現しているか / ライフサイクル
- **定数**: マジックナンバーの由来
- **防御的デフォルト値・非自明な初期化**: なぜここでこの値か

## 書かないもの

### 1. WHAT（コードが既に語ること）

実装を読めば分かることをコメントで重複させない。

- 自明なゲッター/セッター
- フレームワークの定型パターン
- 型名やパラメータ名から明らかなこと

### 2. 歴史・経緯（時間が経つと腐るもの）

| 書かないもの | 例 |
|--------------|----|
| 過去との対比 | 「元は X だったが Y に変えた」「従来は〜」「legacy 〜」「以前の実装と同じルール」 |
| 変更履歴 | 「2024 年 3 月に導入」「〇〇対応で追加」 |
| 個人名・担当者名 | 「〇〇さんが書いた」「△△担当のロジック」 |
| コミット SHA / PR 番号への参照 | 「9b4909a と同じ方式」「PR #1234 で導入」 |

これらは **git log / PR description / 設計ドキュメント** の仕事。コードコメントに混ぜると、前の状態を知らない読者には「そんなん知らんがな」なノイズになり、履歴が進むと陳腐化する。
